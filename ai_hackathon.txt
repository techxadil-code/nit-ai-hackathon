AI/ML HACKATHON — PROBLEM STATEMENT 
Smart Chargesheet Review & Summarisation Assistant 
  Team Size: Max 3 Members 
 
1. Background 
Police Chargesheets (Case Diaries / ककककक कककककक) in India are lengthy multi-page documents written in Hindi. They 
contain incident narratives, witness statements, seizure lists, and legal sections — often spanning 40+ pages. 
Two problems make these documents hard to work with: 
– Reviewing them manually is time-consuming for police officers and public prosecutors. 
– Critical procedural documents (FSL reports, MLC certificates, sanction orders) are frequently missing — which can 
weaken a case during trial. 
 
Mission 
Build an AI assistant that ingests raw Hindi chargesheet text, generates a structured case summary, classifies 
the crime type, and flags missing procedural documents — all through a programmatic API pipeline. 
2. Input Data 
You will be given messy text from real Hindi Case Diaries. Expect: 
– OCR errors and broken character sequences 
– Broken two-column tables rendered as flat unstructured text 
– Code-mixed Hindi/English content (e.g., "SANDISK 32 GB कककक कककक ककक") 
– Inconsistent whitespace and non-standard date formats 
 
Your pipeline must handle this noise. Pre-process intelligently — do not rely on the LLM to fix raw OCR garbage.

4. Stages 
There are two stages. Stage 1 is required. Stage 2 is optional bonus work — attempt it only after Stage 1 is solid. 
 
Stage 1 — Core Pipeline  (Required) 
Given raw Hindi chargesheet text, your system must produce three outputs: 
 
Output A — Structured Case Summary 
– FIR number, date, Police Station name 
– Accused name(s) and Victim name(s) 
– Comprehensive incident facts  
– Legal sections applied (e.g., IPC 379, NDPS 20) 
 
Output B — Crime Classification 
– Classify into one of: Theft/Robbery, Assault/Hurt, Cyber Fraud, NDPS 
– If no confident match, return a graceful fallback: 
{ "crime_type": "UNKNOWN", "reason": "No matching sections found" } 
 
Output C — Missing Items Checklist 
Compare extracted content against the JSON schema for the detected crime type. For each required item, 
output: 
– ✅ PRESENT — evidence clearly found in the text and give its detail in brief 
– ❌ MISSING — item not detected 
– ⚠ PARTIAL — item referenced but appears incomplete/ where it’s referenced and what’s the missing 
part 
 
Technical Requirements 
– All LLM calls must be programmatic via API — no manual copy-pasting into a chat UI. 
– Pre-process noisy text before sending to the LLM (normalize whitespace, strip OCR garbage, detect 
language blocks). 
– Use structured output prompting — ask the LLM to return JSON, then validate it in your code. 
– (optional) Try using the LLM as an extraction engine. Your own deterministic code do the checklist 
comparison against the JSON — not the LLM. 
 
Stage 2 — Bonus: Smarter Extraction  (Optional) 

Attempt this only after Stage 1 is fully working. You can do 2A, 2B, or both — each is independent. 
 
2A — Named Entity Recognition 
Build an NER layer that extracts structured legal entities from the chargesheet text. Entity types to extract: 
 
PERSON 
Accused, victim, witness, officer names 
LEGAL_SECTION 
IPC 379, NDPS 20, IT Act 66C, etc. 
DATE_TIME 
Date of occurrence, FIR date, arrest date 
LOCATION 
Crime scene address, police station name 
DOCUMENT 
FIR, MLC report, seizure memo, FSL report 
AMOUNT 
Stolen property value, drug quantity 
 
Minimum output format: 
{ "entities": [ 
    { "text": "ककक ककककक", "type": "PERSON", "role": "ACCUSED" }, 
    { "text": "IPC 379",   "type": "LEGAL_SECTION" }, 
    { "text": "12/03/2024","type": "DATE_TIME", "event": "FIR_DATE" } 
  ] 
} 
 
2B — Semantic Similarity Scoring 
Stage 1 likely uses keyword matching to detect checklist items. But what if the chargesheet says "ककककककक 
ककककक कककक ककक" instead of the exact phrase "Weighment panchnama"? A keyword check fails. A 
semantic check succeeds. 
 
Replace or augment your Stage 1 checklist detection with a similarity-based approach: 
– Represent each required_item as a vector using TF-IDF or sentence embeddings (e.g., Gemini 
embedding API or any open-source model). 
– Represent each sentence/paragraph of the chargesheet as a vector in the same space. 
– Compute cosine similarity for each required_item against all chargesheet sentences. Use a threshold (e.g., 
0.65) to determine PRESENT / MISSING. 
– Display the top-matching sentence alongside each PRESENT item. 
 
Minimum output format: 
{ "checklist": [ 
    { 
      "item":             "Weighment panchnama", 
      "status":           "PRESENT", 
      "similarity_score": 0.81, 
      "matched_text":     "ककककककक ककककक कककक ककक कककककक 10/03/2024" 
    } 
  ] 
}
